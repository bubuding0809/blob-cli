blob-cli viewer pivot

Adding a tiny user-deployed viewer so private blobs render inline

spec — 2026-04-28 — supersedes parts of v0.1 design

Why pivot

  • blocker Vercel Blob forces Content-Disposition: attachment for HTML, Markdown, most text types — files always download
  • no override No SDK option to set Content-Disposition; verified across @vercel/blob 0.27 and 2.3
  • no escape Vercel Blob has no presigned-download-URL feature — private sharing requires a server-side proxy
  • still useful CLI works fine for inline-eligible types (images, PDFs, plain text, JSON, XML, A/V) — but fails the original "AI emits HTML, human views URL" intent

Architecture

                 ┌──────────────────┐
                 │ user's terminal  │
                 │   (blob upload)  │
                 └────────┬─────────┘
                          │ put / list / delete via SDK
                          ▼
            ┌─────────────────────────────┐
            │  Vercel Blob (private)      │
            │  user-owned store           │
            └─────────────▲───────────────┘
                          │ get(pathname, {access:"private"})
                          │   server-side, with token
        ┌─────────────────┴─────────────────────────┐
        │  Viewer (Vercel deployment)               │
        │  • GET /login,  /api/login → cookie       │
        │  • GET /        → password-gated browser  │
        │  • GET /:path   → public, inline render   │
        │  • GET /api/health                        │
        └─────────────────▲─────────────────────────┘
                          │ HTTPS
                     ┌────┴────┐
                     │ browser │
                     └─────────┘

Three artifacts, all user-owned. No shared backend. Tokens never leave the user's Vercel account.

Auth model — split

RouteAuthWhy
GET / — directory password Owner-only browsing; full file list is enumerable
GET /login, POST /api/login public The login flow itself
GET /api/health public CLI blob init uses for validation
GET /:pathname — file public Shareable links; security via 22-char random suffix
Mental model: / is your private dashboard. Individual file URLs are sharable links — recipients click and see rendered HTML, no login.

Viewer — file proxy


import { get } from "@vercel/blob";

export async function GET(_req, { params }) {
  const pathname = params.pathname.join("/");
  const result = await get(pathname, {
    access: "private",
    token: process.env.BLOB_READ_WRITE_TOKEN!,
  });
  if (!result || result.statusCode !== 200) {
    return new Response("Not found", { status: 404 });
  }
  const ct = result.headers.get("content-type") ?? "application/octet-stream";
  return new Response(result.stream, {
    headers: {
      "content-type": ct,
      "content-disposition": `inline; filename="${pathname.split("/").pop()}"`,
      "cache-control": "private, max-age=300",
    },
  });
}

Override of content-disposition is the whole point — Vercel's CDN otherwise returns attachment and browsers download.

Required env vars (viewer)

VarPurpose

Deploy button in viewer/README.md takes the user to Vercel with root-directory=viewer and prompts for all three envs. ~30s to live URL.

Testing strategy

CLI

  • Existing TDD pattern preserved
  • New validateViewer unit tests
  • init.test — second prompt + retries
  • upload — assert access: "private"
  • SDK 2.3.3 mock signature touchups

Viewer

  • Unit: session.ts sign/verify; tampered/expired rejected
  • Routes: handlers called with mock Request
  • Dashboard: cookie redirect, list() render
  • Manual smoke: deploy → upload → view inline

Out of scope (v1)

  • defer Time-limited / per-recipient signed share URLs
  • defer Browser-side upload / rename / delete from dashboard
  • defer Pagination, search, folder hierarchy
  • defer Auto-deploying the viewer from the CLI
  • no Hosted/shared multi-tenant viewer
  • no Multi-store support in one CLI install

If any of these surface as pressure during implementation, stop and re-spec.
