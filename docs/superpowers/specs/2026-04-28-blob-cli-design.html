blob-cli

A CLI for the agent to publish static files to a public URL — fast upload, no rebuilds, viewable from any browser.

The Problem

Long-form agent output is hard to read in a terminal.

  • Reports, analyses, and dashboards land as walls of markdown.
  • The user has trouble consuming long markdown text in the CLI.
  • The agent can already write HTML — but there's no fast path to view it.

What "fast" means

  • One shell command → public URL.
  • Round-trip in seconds. No app rebuild. No deploy step.

Goals & Non-goals

● Goals

  • One short command → URL.
  • Full CRUD: upload, list, get, delete.
  • npm i -g blob-cli works for anyone — no Bun required.
  • --json output mode so the agent can parse it.

● Non-goals (v1)

  • No private / signed URLs. Everything public.
  • No web UI. CLI is the only interface.
  • No multi-user auth. One token, one user.
  • No file expiry / TTL. You delete when you want.
  • No central server — BYOB.

Implication: don't upload secrets. The only protection on a public URL is the random suffix.

Storage: Vercel Blob

OptionSetupPublic URLVerdict
Vercel Blob Create store, copy token. Done. Returned by put() directly Picked
Supabase Storage Project + bucket + (often) RLS Constructed from bucket + path Fine, but you don't need a DB
S3 / R2 IAM + public-bucket policy + custom domain Assemble it yourself Overkill for this

Free tier: 1 GB storage, generous bandwidth. One env var: BLOB_READ_WRITE_TOKEN.

Onboarding — BYOB

Bring Your Own Blob. The CLI is a thin client; users provide their own Vercel store and token.

● Why this works

  • Zero ops on the maintainer side — no servers, no costs.
  • Privacy by construction: we never touch user files.
  • Free Vercel tier (1 GB / generous bandwidth) is enough.
  • Familiar shape — same as vercel, gh, aws.

● The cost

  • One-time setup: account + store + token.
  • Mitigated by blob init opening the right page and walking through it (next slide).

Hosted-shared model (we run a store, users get accounts on us) is a separate, much larger project — explicitly out of scope.

Runtime & Distribution

Dev

  • Bun runtime — native TS, fast iteration.
  • Source written runtime-agnostically (no Bun.* APIs).
  • bun test for tests.

Distribution

  • bun build --target=node → single bundled JS.
  • Shebang #!/usr/bin/env node.
  • npm package, bin: { "blob": "dist/cli.js" }.
  • npx blob works for everyone.

Bun DX for the developer. Node compat for end users. Both win.

Commands

All commands take --json for machine-readable output.

blob init                              # onboarding flow (next slide)
blob upload <path> [--name N]          # → prints public URL on stdout
blob list   [--prefix P] [--limit N]   # one line per file
blob get    <url-or-pathname> [--out F] # streams to stdout or --out
blob delete <url-or-pathname>          # → "deleted: <url>"

Token resolution order at runtime

  1. BLOB_READ_WRITE_TOKEN env var
  2. ~/.config/blob-cli/config.json
  3. Error pointing at blob init

Behavior

  • Opens the Vercel page via open / xdg-open / start.
  • Validates the token with a list({ limit: 1 }) call — catches typos at init time, not first upload. Up to 3 retries.
  • Idempotent: re-running prompts before overwriting (--force to skip).
  • If BLOB_READ_WRITE_TOKEN env var is set, init notices and exits unless --force.

upload — the hot path

$ blob upload report.html
https://<store>.public.blob.vercel-storage.com/report-x7Ka2.html

$ blob upload report.html --json
{"url":"https://<store>.public.blob.vercel-storage.com/report-x7Ka2.html"}

What it does

  • Reads file at <path>.
  • Calls put(name, content, { access: 'public', addRandomSuffix: true }).
  • --name defaults to file's basename. Random suffix prevents collisions.
  • Content type inferred from extension (text/html, application/json, …); falls back to application/octet-stream.
  • Prints URL to stdout, nothing else (so the agent can capture it cleanly).

Each command file = one function. cli.ts only parses args and dispatches. Small units, easy to test in isolation.

Error Handling

  • Missing token: one-line error referencing blob init, exit 1.
  • Network / 4xx / 5xx: surface SDK error message, exit 1. No retries — the agent can re-invoke if it wants to.
  • File not found on upload: standard ENOENT, exit 1.
  • --json mode: errors as { "error": "..." } on stderr, same exit codes.

Principle: fail loud, fail fast. The caller is an agent — it can decide whether to retry, not us.

Testing

  • Unit tests for config.ts (token resolution precedence) and output.ts (formatting).
  • Integration tests behind a BLOB_TEST_TOKEN env var: real upload → list → get → delete cycle against a real Blob store. Skipped when token is unset.
  • No SDK mocking. The Blob SDK surface is small, and mocks would mask the only failure modes worth catching: auth and network.

For your review

Things worth pushing back on if you don't agree:

  1. BYOB. Each user runs blob init against their own Vercel store. No central server. Confirmed direction?
  2. Public-only. Uploaded files are world-readable to anyone with the URL. OK?
  3. Random-suffix filenames. Every upload gets a unique URL — no "stable URL for the latest report." Acceptable?
  4. Bun → Node bundle. Bun DX during dev; published artifact runs on stock Node. Acceptable compromise?
  5. No retries. Errors propagate up, agent decides. OK or want backoff baked in?
  6. Three deps (@vercel/blob, commander, mime-types). Cut or swap any?

Approve as-is, or tell me what to change before I move to the implementation plan.
